# Бэкенд-сервис для лендинг-презентации разработчика с AI-генерацией текста

## Как запустить проект:

Перед запуском приложения нужно создать файл окружения "./src/.env". Это можно сделать путем копирования .env.example:
`cp ./src/.env.example ./src/.env`

В новом созданном файле нужно будет указать свой токен доступа к OpenRouter (OPENROUTER_API_KEY).

Поднимаем docker-контейнер с помощью команды:
`docker compose up -d --build`

Далее устанавливаем все нужные зависимости composer:
`docker compose exec app composer install`

Генерируем ключ приложения:
`docker compose exec app php artisan key:generate`

Фронтенд-часть приложения будет доступна по адресу:
> http://localhost:8080/

А бэкенд по адресу:
> http://localhost:8081/

**Важно!** При возникновении проблем с правами доступа нужно убедиться, что у пользователя, от лица которого запускается docker, есть доступ к директории ./src/storage

`sudo chmod -R 775 src/storage`

Автоматические тесты можно запустить с помощью команды:
`docker compose exec app php artisan test --group=Feature`

## Архитектура:

**Структура проекта:**

app/Http/Api - API-клиенты для взаимодействия со сторонними сервисами

app/Http/Controllers/Api - контроллеры для API-запросов 

app/Http/Mail - mailable-классы для отправки писем на электронную почту

app/Http/Repositories - репозитории для получения данных из хранилищ

app/Http/Mail/Services - основные сервисы приложения, реализующие бизнес-логику (а также DTO-классы для передачи данных в них)

routes/api.php - регистрация маршрутов на эндпоинты API

storage/app/contact_metrics.json - файл для хранения метрик приложения

**Паттерны проектирования**

Service Layer (Сервисный слой) — Вся бизнес-логика вынесена из контроллеров в отдельные классы-сервисы. Это делает код модульным, чистым и легко тестируемым.

Dependency Injection (Внедрение зависимостей) — Сервисы автоматически внедряются в контроллеры через Service Container Laravel (конструкторы или аргументы методов). Это исключает жесткую связанность кода (tight coupling).

Facade (Фасады) — Читаемый доступ к системным возможностям Laravel (например, Log::info() для логирования).

DTO (Data Transfer Object) — Простые объекты с данными для передачи их между слоями приложения, инкапсуляции и упорядочивания.

Repository (Репозиторий) - Посредник между бизнес-логикой и источником данных (в нашем случае - файловым хранилищем). Он скрывает детали получения информации, отделяя бизнес-логику от логики получениях данных.

    
**Объяснение выбора технологий**

Laravel (PHP 8.x) — Выбран благодаря встроенной экосистеме (удобная система валидации запросов и логирование, удобный роутинг (API Routing) и встроенная обработка ошибок) и относительной простоте/высокой скорости разработки

Отказ от СУБД (использование файлового хранилища) — Операции чтения/записи JSON-файла происходят мгновенно в рамках небольшого объема данных и не требует настройки и администрирования отдельных контейнеров баз данных.

Docker & Docker Compose — Используется для полной изоляции фронтенда и бэкенда, позволяет развернуть и протестировать проект локально без необходимости аренды VPS-сервера.

Mailpit -  Локальный SMTP-сервер и инструмент для тестирования почты. Он выбран, чтобы гарантировать полную автономность проекта в Docker-окружении: проверяющему не нужно вводить реальные пароли от почтовых ящиков, а всю историю отправленных с формы писем можно наглядно посмотреть в веб-интерфейсе Mailpit.

OpenRouter -  Выбран как единый и гибкий шлюз к современным LLM (Large Language Models). Это избавляет от необходимости привязываться к одному провайдеру (например, только OpenAI) и позволяет переключать модели (на более дешевые или мощные) простым изменением переменной в .env файле без правки исходного кода.

## AI-интеграция:
В проекте используются простые бесплатные модели для генерации текста на основе текстовых запросов, предоставляемые сервисом OpenRouter.
В случае, если по какой-то причине сервис недоступен, на почту придет заготовленное шаблонное сообщение, а на клиент придет ответ со статусом 200.
Используемый промпт - просьба классифицировать эмоциональную нагрузку сообщения пользователя (негативная, позитивная, нейтральная) и дать вежливый ответ от лица AI-ассистента разработчика.

## Что сделано с помощью AI:
По большей части - общие шаблоны файлов для настройки Docker и nginx, базовая бизнес-логика отдельных классов.
Промпты по большей части содержали конкретные требования к бизнес-логике с учетом архитектурных нюансов проекта, либо просьбу внести правки, если требовалось.
Вручную приходилось дорабатывать некоторые несоответствия бизнес-логике и архитектуре, менять нейминг и комментарии.

## Хранение данных:
Логи хранятся в файле логов.
Rate limiting реализован с помощью встроенного middleware Laravel (настраивается через файл .env).
Статистика хранится в файле src/storage/app/contact_metrics.json.
